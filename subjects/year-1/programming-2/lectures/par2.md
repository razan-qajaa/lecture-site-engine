## الجزء الأول: الشرح التفصيلي (سطر بسطر / فقرة بفقرة)

> **تصنيف المحاضرة:** محاضرة عن **مؤشرات** → سنستخدم مصطلحات: `&`, `*`, `new`, `delete`, `unique_ptr`, `shared_ptr`

### 1. المصفوفات والمؤشرات

#### النص الأصلي يقول:
> "في لغة `C++` يمكننا التعامل مع المصفوفات على أنها مؤشرات"

#### الشرح المبسّط:
في `C++` اسم المصفوفة نفسه هو في الحقيقة **عنوان أول عنصر فيها**. يعني عندما تكتب `int a[5];` ثم تكتب `int *aPtr = a;`، فأنت لا تخزّن "المصفوفة كاملة" داخل `aPtr`، بل تُخزّن فقط عنوان الخانة الأولى منها.

**لماذا؟** لأن المصفوفة في الذاكرة هي ببساطة سلسلة متتالية من الخانات (Contiguous Memory)، ولا حاجة لتخزين "كل" المصفوفة داخل مؤشر — يكفي معرفة أين تبدأ، وبما أننا نعرف نوع البيانات (`int` مثلاً) نستطيع حساب أي عنصر لاحق.

#### 💡 التشبيه:
> تخيل شارعاً فيه بيوت مرقّمة بالتتالي. لا تحتاج لحفظ عنوان كل بيت على حدة، يكفي أن تعرف عنوان **البيت الأول** ورقم البيت الذي تريده لتصل إليه بالحساب.
> **وجه الشبه:** عنوان البيت الأول = `a` (اسم المصفوفة) → رقم البيت = الإزاحة (index).

```cpp
int main(){
    int a[5];          // (1) array of 5 integers
    int *aPtr = a;      // (2) aPtr now holds the address of a[0]
    cout << a << endl;      // (3) prints address of first element
    cout << *aPtr << endl;  // (4) dereference aPtr -> value of a[0]
    cout << &a[0] << endl;  // (5) address of first element explicitly
}
```

#### شرح كل سطر:
1. `int a[5];` → تعريف مصفوفة — تحجز 5 خانات متتالية من نوع `int`
2. `int *aPtr = a;` → إسناد — `aPtr` يأخذ عنوان أول عنصر (`&a[0]`) تلقائياً دون الحاجة لـ `&`
3. `cout << a << endl;` → طباعة — يطبع عنوان أول عنصر لأن اسم المصفوفة يُقيَّم كمؤشر
4. `cout << *aPtr << endl;` → فك مرجعية — يطبع القيمة المخزنة في العنوان الذي يشير إليه `aPtr`
5. `cout << &a[0] << endl;` → عنونة — يطبع نفس العنوان الظاهر في السطرين السابقين

> **الناتج المتوقع:** الأسطر الثلاثة تطبع **نفس العنوان بالضبط**.

#### نقطة مهمة ⚠️:
> لهذا السبب **لا يتم تمرير المصفوفة بالقيمة (`by value`)** عند تمريرها لتابع، بل يتم تمرير **عنوان أول عنصر فيها فقط** — وهذا مختلف عن باقي المتحولات العادية التي تُمرَّر بالقيمة افتراضياً.

### 2. التعامل مع المصفوفات كمؤشرات والعكس

#### النص الأصلي يقول:
> "تسمح لغة `C++` بالتعامل مع المصفوفات كمؤشر" — مع مثالين متكافئين تماماً في الناتج.

#### الشرح المبسّط:
يمكنك الوصول لعنصر في المصفوفة بطريقتين متكافئتين:
- الطريقة الكلاسيكية: `a[i]`
- طريقة المؤشرات: `*(a + i)`

كلاهما ينتج **نفس القيمة تماماً**، لأن `a[i]` هي في الأساس اختصار كتبه مصمّمو اللغة لعملية `*(a + i)`.

```cpp
int main(){
    int a[5] = {1,2,3,4,5};
    for(int i = 0; i < 5; i++)        // (1) loop over array using index
        cout << a[i] << endl;          // (2) classic array access
    return 0;
}
```
```cpp
int main(){
    int a[5] = {1,2,3,4,5};
    for(int i = 0; i < 5; i++)         // (1) loop over array using index
        cout << *(a+i) << endl;         // (2) pointer arithmetic access
    return 0;
}
```

#### شرح كل سطر:
1. `for(int i=0;i<5;i++)` → حلقة تكرارية تمر على كل عناصر المصفوفة
2. `cout << a[i]` / `cout << *(a+i)` → طريقتان متكافئتان تماماً للوصول لنفس العنصر

#### 🤔 تفعيل الفهم:
> **سؤال:** إذا كانت `a` من نوع `int`، وحجم `int` = 4 بايت، فإن `a+1` تنتقل كم بايت في الذاكرة الفعلية؟
> **لماذا هذا مهم؟** لأنه سيقودنا لأهم فكرة في هذا القسم: **حساب المؤشرات لا يُحسب بالبايت المباشر**.

#### النص الأصلي يقول:
> "نلاحظ من المثال الأيمن أن عملية جمع قيمة للمؤشر لا تسبب الانتقال بايت بايت وإنما خانة خانة بغض النظر عن حجم الخانة"

#### الشرح المبسّط:
عندما نكتب `aPtr + 1`، فإن المؤشر لا يتقدّم بايتاً واحداً، بل يتقدّم **بحجم النوع الذي يشير إليه بالكامل**. فإذا كان `aPtr` من نوع `int*` (و`int` حجمه 4 بايت على أغلب الأنظمة)، فإن `aPtr + 1` تعني: **انتقل 4 بايت للأمام**، وليس بايتاً واحداً.

**لماذا؟** لأن الهدف من `aPtr + 1` هو "أعطني العنصر **التالي**"، وليس "أعطني البايت التالي". لو كانت الإزاحة بايتاً واحداً دائماً، لكان الوصول لعناصر مصفوفة من نوع `int` أو `double` خاطئاً تماماً (لأنها أكبر من بايت واحد).

```cpp
cout << "&a[0]=" << &a[0] << "\n" << "&a[1]=" << &a[1] << endl;
```

> **الناتج المتوقع:**
> ```
> &a[0]=0x7ffd2b4fc960
> &a[1]=0x7ffd2b4fc964
> ```

#### 🔍 تتبع التنفيذ: حساب عنوان العنصر التالي

**المدخل:** `&a[0] = 0x...960` (مصفوفة من نوع `int`، حجم `int` = 4 بايت)

| الخطوة | العملية | الحالة |
| --- | --- | --- |
| 1 | معرفة عنوان `a[0]` | `0x...960` |
| 2 | جمع 1 خانة (`a[1]`) | `0x...960 + 4 بايت = 0x...964` |

**النتيجة:** الفرق بين العنوانين = 4 (بالست عشري: `964 - 960 = 4`)، أي حجم `int` بالضبط وليس 1.

#### 💡 التشبيه:
> تخيل درجاً كل درجة منه ارتفاعها 4 سم. عندما تصعد "درجة واحدة" فإنك تصعد 4 سم فعلياً، وليس 1 سم.
> **وجه الشبه:** رقم الدرجة = الإزاحة (index) | ارتفاع الدرجة = حجم النوع (`sizeof(type)`) | المسافة الفعلية = العنوان الجديد.

### 3. المثال المحذوف من المحاضرة السابقة (ترتيب المتحولات في الذاكرة)

#### النص الأصلي يقول:
> عند تنفيذ الكود التالي على الحاسب بالاعتماد على `G++` نلاحظ أن القيمة المطبوعة هي 50 (أي قيمة المتحول `z`)، بينما عند التنفيذ على `CXXDroid` تكون النتيجة قيمة عشوائية.

```cpp
int main(){
    int x = 10;      // (1) first declared variable
    int z = 50;      // (2) second declared variable
    int *y = &x;     // (3) y points to x
    y++;             // (4) advance pointer by one int-sized step
    cout << *y << endl; // (5) dereference the advanced pointer
}
```

#### شرح كل سطر:
1. `int x=10;` → تعريف أول متحول
2. `int z=50;` → تعريف ثاني متحول
3. `int *y=&x;` → `y` يشير الآن إلى `x`
4. `y++;` → `y` يتقدّم خانة واحدة (بحجم `int`) — **إلى أين بالضبط؟ هذا يعتمد على ترتيب المتحولات في الذاكرة**
5. `cout << *y << endl;` → طباعة ما يوجد في الخانة التي أصبح `y` يشير إليها بعد `y++`

#### الشرح المبسّط:
النتيجة هنا **ليست موحّدة** بين كل المُصرّفات (Compilers)! السبب أن قسم البيانات (`Data Segment`) يُبنى بترتيب مختلف حسب المصرّف:
- في `G++`: يضع المتحولات المُعرَّفة بترتيب **متسلسل ومتتالي**، بحيث تكون خانة `z` **تلي مباشرة** خانة `x` — لذلك `y++` (بعد أن كان يشير لـ `x`) يصل فعلياً إلى `z`، فتُطبع القيمة 50.
- في `CXXDroid`: الترتيب **عكسي**، فتصل النتيجة لخانة عشوائية غير محددة (undefined behavior).

#### مهم للامتحان ⚠️:
> هذا مثال على **Undefined Behavior**: الاعتماد على ترتيب المتحولات في الذاكرة أمر **غير مضمون بالمواصفة القياسية للغة**، ويختلف من مصرّف لآخر. لا يجوز الاعتماد على هذا السلوك في برنامج حقيقي.

#### الفهم الخاطئ الشائع ❌: "زيادة المؤشر بمقدار 1 تعطي نتيجة ثابتة ويمكن التنبؤ بها دوماً"
#### الفهم الصحيح ✅: النتيجة تعتمد على ترتيب حجز المتحولات في الذاكرة، وهذا يختلف حسب المصرّف ولا يجوز الاعتماد عليه

### 3.1 مثال إضافي: y-- مقابل y++

```cpp
// Code that gives 50 on desktop (G++)
int main(){
    int x=10;
    int z=50;
    int *y=&x;
    y++;
    cout<<*y<<endl;
}
```
```cpp
// Code that gives 50 on mobile (CXXDroid)
int main(){
    int x=10;
    int z=50;
    int *y=&x;
    y--;
    cout<<*y<<endl;
}
```

#### الشرح المبسّط:
على الحاسب: `&x=0x...ed8`, `&z=0x...edc` → `z` تأتي بعد `x` في الذاكرة → `y++` يصل لـ `z`.
على الموبايل: `&x=0x...a18`, `&z=0x...a14` → `z` تأتي **قبل** `x` → لذلك يجب استخدام `y--` (وليس `y++`) للوصول لـ `z`.

#### ⚖️ المقايضة: الاعتماد على ترتيب المتحولات في الذاكرة

| | الاعتماد على الترتيب | عدم الاعتماد على الترتيب |
| --- | --- | --- |
| المزايا | لا يوجد فعلياً — مجرد ملاحظة أكاديمية | كود صحيح ومحمول (Portable) على كل الأنظمة |
| العيوب | سلوك غير مُعرَّف (UB)، ينهار عند تغيير المصرّف | لا يوجد |
| متى تختاره | لا تختره أبداً في برنامج حقيقي | دائماً — هذه هي الممارسة الصحيحة |

### 4. التعامل مع المؤشر كمصفوفة

#### النص الأصلي يقول:
> "يمكنك أيضاً التعامل مع المؤشر كأنه مصفوفة"

#### الشرح المبسّط:
تماماً كما استخدمنا `*(a+i)` بدلاً من `a[i]`، يمكن أيضاً أن نأخذ **مؤشراً عادياً** (وليس اسم مصفوفة) ونستخدم معه صيغة `[]` كأنه مصفوفة:

```cpp
int main(){
    int a[50] = {1,2,3,4,5};
    int *b = a;                       // (1) b points to a[0]
    for(int i=0; i<5; i++)            // (2) loop over first 5 elements
        cout << b[i] << endl;          // (3) access via pointer using [] syntax
    return 0;
}
```

#### شرح كل سطر:
1. `int *b = a;` → `b` مؤشر عادي يحمل عنوان أول عنصر من `a`
2. `for(...)` → حلقة على أول 5 عناصر
3. `cout << b[i]` → استخدام `[]` مع **مؤشر عادي** — صحيح تماماً لأن `b[i]` تعني `*(b+i)`

#### نقطة مهمة ⚠️:
> إذن العلاقة **مزدوجة الاتجاه**: المصفوفة تتصرف كمؤشر (`a` ≈ `&a[0]`)، والمؤشر يتصرف كمصفوفة (`b[i]` ≈ `*(b+i)`).

### 5. مخاطر المؤشرات (Buffer Overflow)

#### النص الأصلي يقول:
> "يمكن النظر لقسم البيانات على أنه مصفوفة كبيرة من القيم، يمكن التنقل على هذه القيم باستخدام المؤشرات، هذا أحد أكبر المخاطر التي تعاني منها المؤشرات"

#### الشرح المبسّط:
بما أن المؤشر مجرد عنوان، ولا يوجد فحص تلقائي في `C++` يمنعك من تجاوز حدود المصفوفة، فإنك — نظرياً — تستطيع تحريك مؤشر ليصل لأي منطقة أخرى من الذاكرة (متحولات أخرى أو حتى كود التنفيذ نفسه إذا كان المؤشر مؤشراً على تابع)، وهذا **خطر أمني حقيقي**.

**لماذا خطير؟** لأن الوصول خارج حدود المصفوفة المخصصة يعني قراءة أو كتابة في ذاكرة **لا تخصك**، وقد تحتوي بيانات حساسة (كلمات مرور، مفاتيح تشفير) تابعة لجزء آخر من البرنامج.

#### مهم للامتحان ⚠️:
> ثغرة **Heartbleed** الشهيرة ضمن مكتبة `OpenSSL` هددت خوادم ومواقع الإنترنت حول العالم، بسبب عدم التحقق من حجم المصفوفة أثناء التنقل عليها بالاعتماد على مؤشر.

#### ⚖️ المقايضة: التعامل مع تجاوز حدود المصفوفة (C++ مقابل Java)

| | `Java` وأغلب اللغات الحديثة | `C++` |
| --- | --- | --- |
| المزايا | أمان أعلى — يوقف البرنامج فوراً عند الخطأ (`IndexOutOfBoundException`) | أداء أعلى — لا فحص إضافي في كل وصول |
| العيوب | فحص إضافي يكلّف أداءً بسيطاً في كل وصول للمصفوفة | لا حماية — قد يصل لخانة متحول آخر أو خانة فارغة عشوائية دون أي تحذير |
| متى تختاره | عند أولوية الأمان والموثوقية على السرعة القصوى | عند الحاجة لأقصى أداء ممكن مع انضباط برمجي عالٍ من المطوّر |

#### 💡 التشبيه:
> تخيل مصعداً بدون حساس يوقفه عند آخر طابق — إذا استمريت بالضغط على "طابق أعلى"، سيحاول الصعود إلى خارج المبنى!
> **وجه الشبه:** المصفوفة = المبنى بعدد طوابق محدد | المؤشر يتجاوز الحدود = محاولة الصعود لما بعد آخر طابق دون أي حساس إيقاف.

### 6. الحجز الديناميكي للذاكرة — لماذا نحتاجه؟

#### النص الأصلي يقول:
> "يتم حجز الذاكرة بشكل ديناميكي بالاعتماد على المؤشرات باستخدام الكلمة المفتاحية `new`. لنوضح ذلك، نجيب عن السؤال التالي: كيف يمكننا إنشاء مصفوفة يحدد المستخدم حجمها؟"

#### الشرح المبسّط:
حين تكتب `int a[5];`، فأنت تحدد حجم المصفوفة (5) **وقت كتابة الكود** (Compile Time) — وهذا يسمى **حجز ساكن (Static Allocation)**. لكن ماذا لو أردنا أن يحدد **المستخدم** الحجم أثناء تشغيل البرنامج (Runtime)؟ هنا نحتاج **الحجز الديناميكي (Dynamic Allocation)**.

```cpp
int n;
cin >> n;
int a[n] = {0};       // C++ modern versions (VLA-like, not fully standard)
```
```cpp
int n;
cin >> n;
int *a = new int[n];  // Old / standard-safe C++ dynamic allocation
```

#### شرح كل سطر:
1. `int n;` → تعريف متحول يخزّن حجم المصفوفة المطلوب
2. `cin >> n;` → إدخال الحجم من المستخدم أثناء التنفيذ
3. `int *a = new int[n];` → **حجز `n` خانة من نوع `int` ديناميكياً في الذاكرة**، وإرجاع عنوان أول خانة ليُخزَّن في المؤشر `a`

#### نقطة مهمة ⚠️:
> سنركز على أسلوب **الإصدارات القديمة** (`new int[n]`) لأننا سنحتاجه لاحقاً مع الأغراض (Objects/Classes) — فهو الأساس الذي تُبنى عليه المفاهيم القادمة.

#### 💡 التشبيه:
> `int a[5]` أشبه بحجز طاولة ثابتة الحجم (5 كراسي) قبل وصول الضيوف. أما `new int[n]` فهو أشبه بسؤال الضيوف أولاً "كم عددكم؟" ثم إحضار عدد الكراسي المناسب بالضبط.
> **وجه الشبه:** `n` = عدد الضيوف الفعلي | `new int[n]` = طلب كراسٍ بالعدد المطلوب فقط، لا أكثر ولا أقل.

### 7. مثال متكامل على الحجز الديناميكي

#### النص الأصلي يقول:
> مثال كامل لبرنامج يقرأ حجم مصفوفة من المستخدم، يحجز لها ذاكرة ديناميكياً، يملأها، ثم يطبعها.

```cpp
int main(){
    int n;                         // (1) will hold array size
    cin >> n;                      // (2) read size from user
    int *a = new int[n];           // (3) dynamically allocate n integers
    for(int i=0; i<n; i++)         // (4) loop to fill the array
        cin >> a[i];                // (5) read each value into a[i]
    for(int i=0; i<n; i++)         // (6) loop to print the array
        cout << a[i];               // (7) print each value
    return 0;
}
```

#### شرح كل سطر:
1. `int n;` → تعريف متحول لحفظ الحجم — **بدون هذا المتحول لا يمكن معرفة حجم المصفوفة لاحقاً لأننا لا نستطيع استرجاعه من المؤشر نفسه**
2. `cin >> n;` → إدخال الحجم
3. `int *a = new int[n];` → إنشاء مصفوفة جديدة بالاعتماد على المؤشرات، بحجم يحدده المستخدم
4-5. حلقة إدخال القيم عنصراً عنصراً باستخدام `a[i]` (تماماً مثل المصفوفة العادية)
6-7. حلقة طباعة القيم بنفس الطريقة

#### ⚙️ الخطوات / الخوارزمية: إنشاء مصفوفة ديناميكية الحجم

> **ما هدف هذه العملية؟** إنشاء مصفوفة يحدد المستخدم حجمها أثناء تشغيل البرنامج، بدلاً من تحديد حجم ثابت مسبقاً.

```algorithm
1 | تعريف متحول للحجم | int n; | حجز خانة عادية لتخزين رقم الحجم
2 | إدخال الحجم | cin >> n; | يُقرأ الحجم من المستخدم في وقت التشغيل
3 | حجز الذاكرة | new int[n] | يُحجز مساحة بحجم n في الكومة (Heap) ويُعاد عنوانها
4 | تخزين العنوان | int *a = ... | المؤشر a يحمل الآن عنوان أول خانة محجوزة
5 | تعبئة القيم | حلقة for + cin >> a[i] | تُملأ كل خانة بقيمة يُدخلها المستخدم
6 | طباعة القيم | حلقة for + cout << a[i] | تُعرض القيم المخزّنة
```

#### نقاط التنفيذ:
- **يجب** إدخال الحجم `n` **قبل** استدعاء `new int[n]`، وإلا سيُستخدم حجم غير معرّف
- المصفوفة الناتجة عن `new` تُحجز في **الكومة (Heap)**، على خلاف `int a[5]` التي تُحجز في **المكدّس (Stack)**

### 8. تحرير الذاكرة (Deallocation) — الأمر `delete`

#### النص الأصلي يقول:
> "بعد أن انتهينا من العمل مع هذه المصفوفة علينا التخلص منها، أي هدمها بالاعتماد على الأمر `delete`. عند إنشاء مؤشر باستخدام `new` يتم التخلص منه باستخدام `delete`. عند إنشاء مؤشر باستخدام `new[]` يتم التخلص منه باستخدام `delete[]`"

#### الشرح المبسّط:
كل ذاكرة نحجزها يدوياً بـ `new` هي **مسؤوليتنا** نحن كمبرمجين لتحريرها يدوياً أيضاً، على عكس المتحولات العادية التي يتم التخلص منها تلقائياً عند خروجها من نطاق الرؤية (Scope).

**القاعدة المهمة:** الأداة المستخدمة في الحجز تحدد الأداة المستخدمة في التحرير:
- `new` (لكائن مفرد) ↔ `delete`
- `new[]` (لمصفوفة) ↔ `delete[]`

```cpp
int main(){
    int* p1 = new int;                 // (1) allocate single int
    cout << "Allocated P1" << endl;
    cout << "*p1=" << *p1 << endl << "p1=" << p1 << endl;
    delete p1;                          // (2) free single int
    cout << "DeAllocated P1" << endl;
    cout << "*p1=" << *p1 << endl << "p1=" << p1 << endl;  // (3) dangling pointer access

    int* p2 = new int[10];              // (4) allocate array of 10 ints
    cout << "Allocated P2" << endl;
    cout << "*p2=" << *p2 << endl << "p2=" << p2 << endl;
    delete[] p2;                        // (5) free array with delete[]
    cout << "DeAllocated P2" << endl;
    cout << "*p2=" << *p2 << endl << "p2=" << p2 << endl;   // (6) dangling pointer access
}
```

#### شرح كل سطر:
1. `new int;` → حجز خانة واحدة من نوع `int` (بدون تهيئة، القيمة الابتدائية غير محددة عادةً 0 هنا في المثال)
2. `delete p1;` → تحرير الخانة المحجوزة، لكن `p1` نفسه **لا يزال يحمل نفس العنوان القديم**
3. الوصول لـ `*p1` بعد `delete` → **سلوك غير معرّف (Undefined Behavior)** — الذاكرة لم تعد ملكاً للبرنامج
4. `new int[10];` → حجز مصفوفة من 10 عناصر
5. `delete[] p2;` → تحرير المصفوفة كاملة (يجب استخدام `[]` هنا وليس `delete` فقط)
6. نفس مشكلة السطر 3 لكن على `p2`

> **الناتج المتوقع:**
> ```
> AllocatedP1
> *p1=0
> p1=0x56065c408020
> DeAllocatedP1
> *p1=1617282056
> p1=0x56065c408020
> AllocatedP2
> *p2=0
> p2=0x56065c408750
> DeAllocatedP2
> *p2=1617282056
> p2=0x56065c408750
> ```

#### مهم للامتحان ⚠️:
> لاحظ أن قيمة العنوان (`p1`, `p2`) **لا تتغير** بعد `delete` — المؤشر يبقى يشير لنفس المكان (يُسمى الآن **Dangling Pointer** أي "مؤشر مُعلَّق")، لكن محتوى تلك الخانة أصبح غير محدد لأن نظام التشغيل قد يعيد استخدامها لأي غرض آخر.

#### الفهم الخاطئ الشائع ❌: "بعد `delete p;` يصبح `p` تلقائياً `nullptr`"
#### الفهم الصحيح ✅: `delete` لا تُصفّر المؤشر تلقائياً — يجب على المبرمج تصفيره يدوياً (`p = nullptr;`) لتجنب استخدامه بالخطأ لاحقاً

### 9. تسرّب الذاكرة (Memory Leak)

#### النص الأصلي يقول:
> "تعتبر عملية تحرير الذاكرة عند الانتهاء من الاستخدام عملية أساسية. يجب دائماً تحرير الذاكرة (إعادتها لنظام التشغيل) عند الانتهاء من الاستخدام. ماذا سيحدث إن لم أقم بذلك؟ سيحدث تسرب بالذاكرة `Memory Leak`"

#### الشرح المبسّط:
`Memory Leak` هو نوع من الأخطاء البرمجية الناتج عن **عدم تحرير الذاكرة**. عندما ينتهي استخدام المؤشر (يخرج من نطاق الرؤية مثلاً) دون استدعاء `delete`، تبقى الذاكرة **محجوزة للبرنامج** رغم أنها فعلياً غير مستخدمة — لأن المؤشر الوحيد الذي كان يعرف عنوانها قد اختفى، ولا توجد طريقة أخرى للوصول إليها أو تحريرها.

**لماذا خطير؟** تخيل مصفوفة من 1024 عنصر، كل عنصر بحجم 5 ميجابايت — إذا لم تُحرَّر هذه الذاكرة، ستُهدر 5 جيجابايت من الـ `RAM` بلا أي فائدة، وقد يتكرر هذا الخطأ آلاف المرات في برنامج طويل التشغيل (مثل خادم يعمل لأشهر) حتى تنفد الذاكرة بالكامل.

#### 💡 التشبيه:
> تخيل أنك تستأجر خزانة في مستودع عام، وبعد أن تفرغ محتوياتها تنسى إخبار المستودع بأنك انتهيت — ستبقى الخزانة "محجوزة باسمك" رغم أنها فارغة، ولن يستطيع أحد آخر استخدامها.
> **وجه الشبه:** الخزانة = الذاكرة المحجوزة | نسيان إخبار المستودع = نسيان استدعاء `delete` | الخزائن الممتلئة بلا فائدة تتراكم = تسرب الذاكرة التراكمي حتى نفاد المساحة.

### 10. كيف تحل اللغات الأخرى مشكلة تسرب الذاكرة؟

#### النص الأصلي يقول:
> "في لغات `Python`، `C#‎`، `Java` و`JavaScript` يتم الاعتماد على مفهوم جامع القمامة `Garbage Collector` الذي يتولى عملية إدارة الذاكرة بشكل كامل ويقوم بتحرير حجر الذاكرة غير المستخدمة أثناء التنفيذ. في لغة `Rust` تم تطوير آلية لإدارة الذاكرة تعتمد مفهوم الملكية والاستعارة `Ownership & Borrowing` يتم تنفيذها أثناء الترجمة. معناها لغة `C++‎` فاشلة صح؟ Well Yes but actually No"

#### الشرح المبسّط:
- **Garbage Collector** (في `Python`, `C#`, `Java`, `JavaScript`): برنامج يعمل في الخلفية أثناء تشغيل برنامجك (Runtime)، يراقب أي ذاكرة لم تعد مُستخدَمة ويحررها تلقائياً — المبرمج **لا يستدعي** `delete` بنفسه إطلاقاً.
- **Ownership & Borrowing** (في `Rust`): آلية أذكى تُفرض **أثناء الترجمة (Compile Time)** وليس أثناء التشغيل — المُصرّف نفسه يرفض ترجمة الكود إذا اكتشف احتمال تسرب أو استخدام خاطئ للذاكرة، دون أي كلفة أداء إضافية وقت التشغيل.

فهل هذا يعني أن `C++` أسوأ لأنها تعتمد على المبرمج يدوياً؟ الجواب: **ليس بالضرورة**، والسبب هو مجال الاستخدام (راجع القسم التالي).

#### ⚖️ المقايضة: إدارة الذاكرة اليدوية مقابل الآلية

| | إدارة يدوية (`C++` الكلاسيكية: `new`/`delete`) | إدارة آلية (`Garbage Collector`) |
| --- | --- | --- |
| المزايا | تحكم كامل، لا كلفة أداء إضافية وقت التشغيل، مناسبة للأنظمة الحساسة زمنياً | راحة للمبرمج، تقليل احتمال الخطأ البشري |
| العيوب | مسؤولية المبرمج بالكامل — نسيان `delete` = تسرب ذاكرة | كلفة أداء إضافية (فحص دوري)، توقفات غير متوقعة (Pauses) |
| متى تختاره | أنظمة تشغيل، ألعاب، أنظمة تعمل بزمن حقيقي (Real-time) | تطبيقات ويب، تطبيقات عادية لا تتطلب أداءً حرجاً للغاية |

### 11. لماذا لا تزال C++ (ولغات مشابهة) ضرورية؟

#### النص الأصلي يقول:
> "لكي نحدد فشل أم نجاح اللغة علينا مراعاة مجال التطبيق. لغة `JavaScript` تحتاج لبيئة متصفح أو `NodeJS`... لا يمكنها أن تعمل على العتاد مباشرة، يجب أن يوجد نظام تشغيل. `C#‎` بحاجة لتواجد بيئة `.Net`... `Java` بحاجة لتواجد `JVM`... `Python` تعاني من نفس القصة. لكن: `C/C++/Rust` لا تحتاج لتواجد البيئة، أي يمكن تشغيل أكوادها على العتاد مباشرة"

#### الشرح المبسّط:
كل اللغات التي تعتمد على `Garbage Collector` (مثل `Java`, `C#`, `Python`, `JavaScript`) تحتاج **بيئة تشغيل وسيطة** (`JVM`, `.Net Runtime`, `Interpreter`, متصفح...) لتعمل — وهذه البيئة نفسها **يجب أن تكون مكتوبة بلغة تعمل مباشرة على العتاد (Hardware)** بدون أي وسيط، وهنا يأتي دور `C`, `C++`, `Rust`.

**لماذا هذا مهم؟** لأن أنظمة التشغيل، والمتصفحات، وأنظمة المتحكمات المصغّرة (`Microcontroller`) لا يوجد "شيء" أسفلها لتشغيلها عليه — يجب أن تعمل مباشرة على المعالج.

#### النص الأصلي يقول:
> أمثلة: `Microsoft Office (C++)`, `Mozilla Firefox (C++ + Rust)`, `Chrome (C++)`, `Android Kernel (Linux is in C++ + Rust)`, الألعاب التي تعتمد على `Unreal Engine` مكتوبة باستخدام `C++`

#### مهم للامتحان ⚠️:
> الخلاصة: **لا توجد لغة "أفضل" بإطلاق** — كل لغة صُممت لمجال معين. `C++` ضرورية لأنها تُستخدم لبناء **البيئات** التي تعمل ضمنها اللغات الأخرى، بالإضافة لاستخدامها المباشر في البرامج التقليدية عالية الأداء.

### 12. المؤشرات الذكية (Smart Pointers) — الحل الوسط في C++‎

#### النص الأصلي يقول:
> "كل ما سبق لا يعني أن `C++‎` كانت قاعدة ما عم تعمل شي! تم تطوير آلية تسمح بما يشبه الإدارة الآلية للذاكرة الديناميكية بالاعتماد على النمطين الجديدين: `UniquePtr`, `SharedPtr`. هذين النمطين مؤشران ذكيان. `unique` يعتبر مالكاً حصرياً للقيمة. `shared` يسمح بتشارك القيمة"

#### الشرح المبسّط:
المؤشرات الذكية (`Smart Pointers`) هي حل وسط ذكي بين النمط اليدوي البحت (`new`/`delete`) والحل الآلي الكامل (`Garbage Collector`): هي **كائنات (Objects)** تتصرف مثل المؤشرات العادية، لكنها **تتذكر بنفسها** متى يجب حذف البيانات التي تشير إليها، وتقوم بذلك **تلقائياً** دون أن يستدعي المبرمج `delete` يدوياً.

يوجد نمطان رئيسيان:
- **`unique_ptr`**: مالك **حصري** واحد فقط للقيمة، لا يمكن نسخه لمؤشر آخر
- **`shared_ptr`**: يسمح لعدة مؤشرات **بتشارك** ملكية نفس القيمة في آنٍ واحد

#### 💡 التشبيه:
> `unique_ptr` أشبه بمفتاح سيارة شخصية واحد لا يُنسخ — إن أعطيته لشخص آخر، فقدته أنت. أما `shared_ptr` أشبه بمفتاح شقة مشتركة بين عدة أشخاص — الشقة تُغلق فقط عندما يُعيد **آخر شخص** مفتاحه.
> **وجه الشبه:** `unique_ptr` = المفتاح غير القابل للنسخ | `shared_ptr` = المفتاح المشترك مع عداد لعدد حاملي النسخ.

### 13. unique_ptr — المفهوم والخصائص التقنية

#### النص الأصلي يقول:
> "الإدارة الذاتية: هي 'مؤشر ذكي' (`Smart Pointer`) يمتلك كائناً بشكل حصري. يقوم تلقائياً بحذف الكائن وتحرير الذاكرة بمجرد خروج المؤشر عن مجال الرؤية (`Scope`). الملكية الفردية: لا يمكن لنسختين من `unique_ptr` امتلاك نفس الكائن في نفس الوقت."

#### الشرح المبسّط:
`unique_ptr` يحرص على أن يكون هناك **مالك واحد فقط** لكل قطعة ذاكرة في أي لحظة. بمجرد أن يخرج `unique_ptr` من نطاق تعريفه (مثلاً نهاية التابع الذي عُرِّف بداخله)، فإن الذاكرة تُحرَّر **تلقائياً** دون تدخل المبرمج — وهذا يحل مشكلة تسرب الذاكرة الناتجة عن النسيان.

#### الخصائص التقنية:
- **غير قابل للنسخ (`Non-copyable`):** لا يمكنك نسخ `unique_ptr` إلى آخر (لأن ذلك سيؤدي لوجود مالكين لنفس المورد، وهذا يخالف مبدأ "الملكية الحصرية")
- **قابل للنقل (`Movable`):** يمكنك نقل الملكية من مؤشر لآخر باستخدام `std::move`. بعد النقل، يصبح المؤشر القديم فارغاً (`nullptr`)

#### حالات حذف الكائن تلقائياً:
1. عند تدمير كائن `unique_ptr` (خروجه من النطاق)
2. عند تعيين قيمة جديدة للمؤشر باستخدام معامل التعيين (`=`) أو دالة `reset()`

```cpp
#include<iostream>
#include <memory>       // (1) required for smart pointers
using namespace std;
int main(){
    unique_ptr<int> ptr = make_unique<int>();  // (2) create a unique_ptr (not with new!)
    *ptr = 5;                                   // (3) assign value via dereference
    cout << *ptr << endl;                        // (4) print value
}
```

#### شرح كل سطر:
1. `#include <memory>` → المكتبة المطلوبة لاستخدام `unique_ptr` و`shared_ptr`
2. `unique_ptr<int> ptr = make_unique<int>();` → إنشاء مؤشر ذكي جديد **بالاعتماد على `make_unique` وليس الأمر `new`**
3. `*ptr = 5;` → التعامل معه يتم كما يتم التعامل مع أي مؤشر آخر (فك مرجعية بـ `*`)
4. `cout << *ptr << endl;` → طباعة القيمة

**المكتبات المطلوبة (Imports):**
> `#include <memory>`

**الناتج المتوقع:**
> `5`

### 14. unique_ptr والنقل (Move)

```cpp
#include<iostream>
#include <memory>
using namespace std;
int main(){
    unique_ptr<int> ptr = make_unique<int>();
    *ptr = 5;
    cout << *ptr << endl;              // (1) prints 5

    unique_ptr<int> x = move(ptr);     // (2) transfer ownership from ptr to x
    cout << *ptr << endl;               // (3) ERROR: ptr no longer owns anything
    cout << *x;                          // (4) prints 5, x is the new owner
}
```

#### شرح كل سطر:
1. طباعة أولى تعمل بشكل طبيعي، تطبع `5`
2. `unique_ptr<int> x = move(ptr);` → **يتم نقل الملكية** من المؤشر `ptr` إلى المؤشر `x` — `ptr` يصبح فارغاً بعد هذه العملية
3. `cout << *ptr` → **سيحدث خطأ هنا** لأن `ptr` لم يعد يؤشر إلى أي شيء بفعل نقل الملكية إلى `x`
4. `cout << *x` → يعمل بشكل صحيح ويطبع `5`، لأن `x` هو المالك الجديد

#### 🔄 قبل / بعد: عملية `move(ptr)`

**قبل:**
```cpp
unique_ptr<int> ptr = make_unique<int>();  // ptr يملك القيمة 5
```

**بعد:**
```cpp
unique_ptr<int> x = move(ptr);   // x يملك القيمة الآن، ptr أصبح فارغاً (nullptr)
```

**ماذا تغيّر؟** انتقلت الملكية الحصرية من `ptr` إلى `x`، وأصبح `ptr` بلا قيمة صالحة للاستخدام.

#### مهم للامتحان ⚠️:
> محاولة فك مرجعية (`dereference`) مؤشر `unique_ptr` بعد نقل ملكيته (`move`) يُعتبر **خطأ منطقياً** وسلوكاً غير آمن — يجب دائماً التحقق أن المؤشر لا يزال صالحاً قبل استخدامه.

### 15. shared_ptr — الملكية المشتركة

#### النص الأصلي يقول:
> "بخلاف `unique_ptr` الذي يسمح بمالك واحد فقط، تتيح `shared_ptr` لعدة مؤشرات أن تمتلك نفس الكائن في الذاكرة. يتم تدمير الكائن وتحرير الذاكرة فقط عندما يتم تدمير آخر `shared_ptr` يمتلك ذلك الكائن."

#### الشرح المبسّط:
`shared_ptr` يسمح لعدة مؤشرات بالإشارة **لنفس الكائن** في آنٍ واحد. لضمان عدم حذف الكائن مبكراً بينما لا يزال أحدهم يستخدمه، يعتمد `shared_ptr` على آلية **عداد المراجع (Reference Counting)**.

#### آلية العمل — كتلة التحكم (Control Block):
يستخدم `shared_ptr` داخلياً "كتلة تحكم" تتضمن:
- **عداد المراجع (`Shared Counter`):** يحسب عدد المؤشرات التي تمتلك الكائن حالياً
- **عداد المراجع الضعيفة (`Weak Counter`):** يحسب عدد مؤشرات `weak_ptr` المرتبطة به (غير مطلوب في هذه المرحلة)
- **المُدمِّر (`Deleter`):** الطريقة التي سيتم بها حذف الكائن

#### العمليات الأساسية:
- **النسخ والتعيين:** عند نسخ `shared_ptr` إلى آخر، يزداد عداد المراجع بمقدار 1
- **النقل (`Move`):** عند نقل `shared_ptr` إلى آخر، تنتقل الملكية ولا يتغير عداد المراجع (يصبح المؤشر القديم فارغاً)
- **التدمير:** عند خروج المؤشر من المجال، ينقص العداد بمقدار 1؛ إذا وصل العداد إلى صفر، يُحذف الكائن

#### أفضل ممارسات الإنشاء:
> `std::make_shared` هي الطريقة **المفضلة والآمنة** لإنشاء `shared_ptr` — تتميز بكفاءة أعلى لأنها تخصص الذاكرة للكائن **وكتلة التحكم** في عملية تخصيص واحدة (`Single Allocation`).

```cpp
#include<iostream>
#include <memory>
using namespace std;
int main()
{
    shared_ptr<int> ptr = make_shared<int>();  // (1) create shared_ptr, ref count = 1
    *ptr = 5;                                    // (2) assign value
    cout << *ptr << endl;                          // (3) print 5, count still 1
    shared_ptr<int> x = ptr;                     // (4) copy: ref count becomes 2
    cout << *ptr << endl;                          // (5) still valid, prints 5
    cout << *x;                                    // (6) also valid, prints 5
}
```

#### شرح كل سطر:
1. `make_shared<int>()` → إنشاء `shared_ptr` جديد، عداد المراجع = 1
2. `*ptr = 5;` → إسناد قيمة
3. طباعة أولى — تعمل بشكل طبيعي
4. `shared_ptr<int> x = ptr;` → **نسخ** (وليس نقل) — الآن `x` و`ptr` كلاهما يشيران لنفس الكائن، عداد المراجع أصبح 2
5-6. كلا المؤشرين لا يزالان صالحين للاستخدام معاً — على خلاف `unique_ptr` تماماً

**الناتج المتوقع:**
> ```
> 5
> 5
> 5
> ```

#### ⚖️ المقايضة: unique_ptr مقابل shared_ptr

| | `unique_ptr` | `shared_ptr` |
| --- | --- | --- |
| المزايا | كلفة أقل (بدون عداد مراجع)، وضوح تام في الملكية | مرونة أعلى، مناسب عندما يحتاج أكثر من طرف الوصول لنفس المورد |
| العيوب | لا يمكن مشاركة نفس المورد بين أكثر من مالك | كلفة أعلى قليلاً (إدارة عداد المراجع)، احتمال حلقات مرجعية (circular references) بدون `weak_ptr` |
| متى تختاره | عندما يكون واضحاً أن كائناً واحداً فقط يملك المورد طوال حياته | عندما يحتاج عدة أجزاء من البرنامج للوصول لنفس المورد في آنٍ واحد |

#### 🤔 تفعيل الفهم:
> **سؤال:** لماذا لا نستخدم `shared_ptr` دائماً بدلاً من `unique_ptr` كخيار افتراضي آمن؟
> **لماذا هذا مهم؟** لأن `shared_ptr` يحمل كلفة أداء إضافية (عداد مراجع يُحدَّث في كل نسخ/تدمير) لا داعي لها إن كان المورد يخص كائناً واحداً فقط — القاعدة الذهبية: استخدم `unique_ptr` افتراضياً، وانتقل لـ `shared_ptr` فقط عند الحاجة الفعلية للمشاركة.

## الجزء الثاني: ملخص منظم

### أهم التعاريف والمفاهيم

| المصطلح | التعريف | مثال/ملاحظة |
| --- | --- | --- |
| `Pointer Arithmetic` | حساب المؤشرات — إضافة/طرح رقم من مؤشر يعني التنقل بعدد كامل من عناصر النوع | `a+1` تنتقل بحجم `sizeof(type)` وليس بايت واحد |
| `Dynamic Allocation` | حجز ذاكرة أثناء التشغيل (Runtime) بدلاً من وقت الترجمة | `new int[n]` |
| `Static Allocation` | حجز ذاكرة بحجم ثابت معروف وقت الترجمة | `int a[5];` |
| `Heap` | منطقة الذاكرة التي تُحجز فيها الذاكرة الديناميكية | تُدار يدوياً عبر `new`/`delete` |
| `Stack` | منطقة الذاكرة التي تُحجز فيها المتحولات العادية والمصفوفات الثابتة | تُدار تلقائياً عند دخول/خروج النطاق |

| `Garbage Collector` | آلية آلية لتحرير الذاكرة غير المستخدمة أثناء التشغيل | مستخدم في `Java`, `C#‎`, `Python`, `JavaScript` |
| `Ownership & Borrowing` | آلية إدارة ذاكرة تُفرض وقت الترجمة | مستخدمة في لغة `Rust` |
| `Smart Pointer` | مؤشر ذكي يدير حذف الذاكرة تلقائياً | `unique_ptr`, `shared_ptr` |

| `shared_ptr` | مؤشر ذكي مالك مشترك — يعتمد على عداد مراجع | `make_shared<int>()` |
| `Reference Counting` | عدّ عدد المؤشرات المالكة لنفس الكائن | يستخدمه `shared_ptr` داخلياً |
| `Control Block` | كتلة تحكم داخلية في `shared_ptr` تضم عداد المراجع والمُدمِّر | تُنشأ تلقائياً مع `make_shared` |

| `make_unique` | إنشاء `unique_ptr` بأمان | الطريقة الموصى بها بدلاً من `new` المباشر |

| المؤشرات الذكية | `unique_ptr` | `shared_ptr` | ملكية حصرية غير قابلة للنسخ مقابل ملكية مشتركة بعداد مراجع |
| إدارة الذاكرة | `C++‎` الكلاسيكية (`new`/`delete`) | `Java`/`C#‎`/`Python` (`Garbage Collector`) | يدوية بالكامل مقابل آلية أثناء التشغيل |

| الأداء والبيئة | `C/C++/Rust` | `Java/C#‎/Python/JavaScript` | تعمل مباشرة على العتاد مقابل تحتاج بيئة تشغيل وسيطة |

### أبرز النقاط الذهبية
1. اسم المصفوفة يُقيَّم كعنوان أول عنصر فيها؛ لذلك تُمرَّر المصفوفات دائماً بالعنوان لا بالقيمة
2. `a[i]` و `*(a+i)` متكافئتان تماماً — والمؤشر العادي يمكن استخدامه بصيغة `[]` أيضاً
3. حساب المؤشرات (`Pointer Arithmetic`) يتم بوحدة "عنصر كامل" وليس بايت واحد، أي يُضرب بـ `sizeof(type)`
4. `C++‎` لا تتحقق تلقائياً من تجاوز حدود المصفوفة — هذا مصدر مخاطر أمنية حقيقية (مثل Heartbleed)
5. الحجز الديناميكي (`new`) يسمح بتحديد حجم المصفوفة أثناء التشغيل بدلاً من وقت الترجمة
6. كل `new` يجب أن يقابله `delete`، وكل `new[]` يجب أن يقابله `delete[]`
7. نسيان `delete` يؤدي لـ `Memory Leak` — تراكم ذاكرة محجوزة وغير مستخدمة
8. لغات مثل `Java` و`Python` تحل هذه المشكلة بـ `Garbage Collector`، بينما `Rust` تحلها وقت الترجمة عبر `Ownership`
9. `C/C++/Rust` ضرورية لأنها تعمل مباشرة على العتاد دون الحاجة لبيئة تشغيل وسيطة
10. المؤشرات الذكية (`unique_ptr`, `shared_ptr`) توفر حلاً وسطاً يحرر الذاكرة تلقائياً عبر أدوات `C++‎` نفسها دون الحاجة لـ Garbage Collector كامل

### الأخطاء الشائعة عند الطلاب ⚠️

| الخطأ | التصحيح |
| --- | --- |
| الاعتقاد أن `a + 1` تنتقل بايتاً واحداً دائماً | تنتقل بحجم النوع (`sizeof(type)`) وليس بايتاً واحداً |
| استخدام `delete` بدلاً من `delete[]` مع مصفوفة محجوزة بـ `new[]` | يجب مطابقة الأداة: `new[]` ↔ `delete[]` فقط |
| افتراض أن `delete p;` تجعل `p` يساوي `nullptr` تلقائياً | يجب تصفير المؤشر يدوياً بعد `delete` إن أردت تجنب استخدامه بالخطأ |
| نسخ `unique_ptr` مباشرة (`=`) | `unique_ptr` غير قابل للنسخ، استخدم `std::move` للنقل فقط |
| استخدام `*ptr` بعد `move(ptr)` | المؤشر الأصلي أصبح فارغاً بعد النقل، يجب استخدام المؤشر الجديد |
| الاعتماد على ترتيب المتحولات في الذاكرة (Stack layout) في كود حقيقي | هذا سلوك غير معرّف (UB) يختلف حسب المصرّف، لا يجوز الاعتماد عليه |

### خطوات وإجراءات المحاضرة

#### ⚙️ الخطوات / الخوارزمية: إنشاء مصفوفة ديناميكية الحجم وتعبئتها وطباعتها

> **ما هدفه؟** إنشاء مصفوفة بحجم يحدده المستخدم أثناء التشغيل، ثم تعبئتها وعرضها.

```algorithm
1 | تعريف متحول الحجم | int n; | حجز خانة عادية للحجم
2 | إدخال الحجم | cin >> n; | قراءة الحجم من المستخدم
3 | حجز الذاكرة | new int[n] | حجز n خانة في الـ Heap
4 | تعبئة القيم | for + cin >> a[i] | إدخال كل قيمة عنصر بعنصر
5 | طباعة القيم | for + cout << a[i] | عرض القيم المخزّنة
```

#### نقاط التنفيذ:
- يجب إدخال `n` قبل استدعاء `new int[n]`
- عدم نسيان تحرير الذاكرة لاحقاً بـ `delete[] a;`

#### ⚙️ الخطوات / الخوارزمية: تحرير الذاكرة المحجوزة يدوياً

> **ما هدفه؟** إعادة الذاكرة المحجوزة يدوياً إلى نظام التشغيل بعد الانتهاء من استخدامها لمنع تسرب الذاكرة.

```algorithm
1 | تحديد نوع الحجز | new أو new[] | معرفة الأداة المناسبة للتحرير
2 | استدعاء أداة التحرير | delete p; أو delete[] p; | إعادة الذاكرة لنظام التشغيل
3 | (اختياري) تصفير المؤشر | p = nullptr; | تجنب استخدام مؤشر معلّق بالخطأ لاحقاً
```

#### نقاط التنفيذ:
- استخدام `delete` مع ذاكرة محجوزة بـ `new[]` (أو العكس) سلوك غير معرّف
- الوصول للقيمة بعد `delete` مباشرة يُعتبر خطأً منطقياً حتى لو لم يتوقف البرنامج فوراً

#### ⚙️ الخطوات / الخوارزمية: إنشاء واستخدام unique_ptr

> **ما هدفه؟** حجز ذاكرة تُدار تلقائياً بملكية حصرية دون الحاجة لاستدعاء `delete` يدوياً.

```algorithm
1 | تضمين المكتبة | #include <memory> | تفعيل استخدام المؤشرات الذكية
2 | إنشاء المؤشر | make_unique<T>() | إنشاء unique_ptr آمن بدلاً من new مباشرة
3 | الاستخدام | *ptr = value; | التعامل معه كمؤشر عادي
4 | (اختياري) نقل الملكية | unique_ptr<T> x = move(ptr); | ptr يصبح فارغاً بعد النقل
5 | التحرير التلقائي | (يحدث ضمنياً) | يُحذف الكائن تلقائياً عند خروج المؤشر من النطاق
```

#### نقاط التنفيذ:
- لا يمكن نسخ `unique_ptr` بمعامل التعيين المباشر — فقط بـ `std::move`
- بعد `move`، لا تستخدم المؤشر الأصلي مطلقاً

#### ⚙️ الخطوات / الخوارزمية: إنشاء واستخدام shared_ptr

> **ما هدفه؟** السماح لعدة أطراف بمشاركة ملكية نفس الكائن مع تحرير تلقائي عند انتهاء آخر مالك.

```algorithm
1 | تضمين المكتبة | #include <memory> | تفعيل استخدام المؤشرات الذكية
2 | إنشاء المؤشر | make_shared<T>() | إنشاء shared_ptr مع كتلة تحكم واحدة (عداد المراجع = 1)
3 | مشاركة الملكية | shared_ptr<T> x = ptr; | نسخ يزيد عداد المراجع بمقدار 1
4 | التحرير التلقائي | (يحدث ضمنياً) | يُحذف الكائن فقط عند وصول العداد للصفر
```

#### نقاط التنفيذ:
- استخدم `make_shared` دائماً بدلاً من `shared_ptr<T>(new T())` المباشر لأنها أكثر كفاءة
- انتبه لاحتمال الحلقات المرجعية بين عدة `shared_ptr` (يُحل لاحقاً بـ `weak_ptr`)

### أنماط الأكواد والبنى المتكررة

| النمط | البنية الأساسية | متى تستخدمه |
| --- | --- | --- |
| مصفوفة ديناميكية الحجم | `int *a = new int[n];` | عندما لا يُعرف حجم المصفوفة إلا وقت التشغيل |
| تحرير مصفوفة ديناميكية | `delete[] a;` | فوراً بعد الانتهاء من استخدام المصفوفة الديناميكية |
| مؤشر ذكي حصري | `unique_ptr<T> p = make_unique<T>();` | عندما يكون هناك مالك واحد واضح للمورد طوال حياته |
| مؤشر ذكي مشترك | `shared_ptr<T> p = make_shared<T>();` | عندما تحتاج عدة أجزاء من البرنامج نفس المورد في آنٍ واحد |
| نقل ملكية `unique_ptr` | `unique_ptr<T> x = move(p);` | عند الحاجة لتمرير ملكية المورد لمتحول آخر دون نسخ |

### أنماط التعامل والسلوك

| السيناريو | التعامل الصحيح | لماذا؟ |
| --- | --- | --- |
| الحاجة لمصفوفة بحجم غير معروف مسبقاً | استخدام `new int[n]` بعد قراءة `n` | لأن `int a[5]` يتطلب حجماً ثابتاً وقت الترجمة |
| الانتهاء من استخدام ذاكرة محجوزة بـ `new[]` | استدعاء `delete[]` وليس `delete` | لتجنب سلوك غير معرّف ومطابقة أداة التحرير مع أداة الحجز |
| الحاجة لمشاركة كائن بين أكثر من جزء بالبرنامج | استخدام `shared_ptr` بدلاً من مؤشر عادي | لضمان عدم حذف الكائن طالما لا يزال أحد يستخدمه (عداد مراجع) |
| نقل ملكية مورد من متحول لآخر دون نسخ فعلي | استخدام `std::move` مع `unique_ptr` | لأن `unique_ptr` غير قابل للنسخ بطبيعته |

## الجزء الثالث: أسئلة اختيار من متعدد (MCQ)

> **16 سؤالاً** — مستوى: متوسط/صعب. التوزيع: مقارنات 25%، سيناريو كود 35%، تطبيق 30%، تتبع خوارزمية 10%.

### السؤال 1 (سهل)
ما الذي يمثله اسم المصفوفة `a` في التعبير `int *aPtr = a;`؟
أ) قيمة أول عنصر في المصفوفة
ب) عنوان أول عنصر في المصفوفة
ج) عدد عناصر المصفوفة
د) عنوان آخر عنصر في المصفوفة
**الإجابة الصحيحة: ب**
**التعليل:** اسم المصفوفة يُقيَّم كعنوان أول عنصر فيها (`&a[0]`)، وليس كقيمة أو عدد. الخيار (أ) يخلط بين المؤشر والقيمة المخزنة، (ج) يخلط بينه وبين `sizeof`/`n`، (د) خاطئ لأنه يشير للعنصر الأول لا الأخير.

### السؤال 2 (متوسط)
ماذا يطبع الكود التالي إذا كان `int` حجمه 4 بايت و `&a[0] = 0x1000`؟
```cpp
int a[5] = {10,20,30,40,50};
cout << &a[1];
```
أ) `0x1001`  ب) `0x1004`  ج) `0x1002`  د) `0x1008`
**الإجابة الصحيحة: ب**
**التعليل:** الانتقال بمقدار خانة واحدة في مصفوفة `int` ينتقل بحجم `int` بالكامل (4 بايت)، أي `0x1000 + 4 = 0x1004`. باقي الخيارات تفترض خطأً إزاحة بايت واحد أو ضعف الحجم أو نصفه.

### السؤال 3 (سهل)
أي تعبير من التالي **مكافئ تماماً** لـ `a[i]`؟
أ) `a + i`  ب) `*(a+i)`  ج) `*a + i`  د) `&a + i`
**الإجابة الصحيحة: ب**
**التعليل:** `a[i]` هو اختصار لغوي لـ `*(a+i)`. الخيار (أ) يعطي عنواناً وليس قيمة، (ج) يجمع قيمة العنصر الأول مع `i` وهذا خاطئ رياضياً، (د) يستخدم عنوان المصفوفة كاملة وليس عنوان عنصر.

### السؤال 4 (متوسط)
لماذا لا تُمرَّر المصفوفات بالقيمة (`by value`) للتوابع في `C++‎`؟
أ) لأن المصفوفات دائماً فارغة عند التمرير
ب) لأن اسم المصفوفة هو عنوان أول عنصر فيها، فيُمرَّر العنوان فقط
ج) لأن التوابع لا تقبل مصفوفات كمعاملات إطلاقاً
د) لأن حجم المصفوفة يتغير تلقائياً عند التمرير
**الإجابة الصحيحة: ب**
**التعليل:** بما أن اسم المصفوفة يُقيَّم كعنوان، فإن تمريرها للتابع ينقل العنوان فقط (تمرير بالعنوان الفعلي)، وهذا يفسر لماذا التعديل داخل التابع يؤثر على المصفوفة الأصلية. باقي الخيارات غير صحيحة منطقياً وتقنياً.

### السؤال 5 (صعب)
ما سبب اختلاف نتيجة الكود التالي بين `G++‎` و`CXXDroid`؟

| تطبيق ويب داخلي بعدد مستخدمين محدود | إدارة آلية (`Garbage Collector`، مثلاً `C#‎`/`Java`) | لا حاجة لأداء حرج بالمستوى الأقصى، والراحة في التطوير وتقليل الأخطاء أهم |

### تمرين 2: تحليل سيناريو تسرب ذاكرة في خادم طويل التشغيل

**السيناريو:**
خادم ويب يعمل بشكل متواصل لعدة أشهر، ويحجز ذاكرة ديناميكياً لكل طلب مستخدم يصل إليه دون تحريرها بعد الانتهاء من معالجة الطلب.

**المطلوب:**
1. صف ماذا سيحدث للخادم على المدى الطويل
2. اقترح حلاً برمجياً للمشكلة

**نموذج الحل:**
1. مع كل طلب جديد تتراكم ذاكرة إضافية محجوزة وغير محرَّرة (Memory Leak تراكمي)، حتى تنفد ذاكرة الخادم بالكامل ويتوقف عن العمل أو يتباطأ بشدة (تحديداً مشكلة شائعة في الخوادم طويلة الأمد Long-running services)
2. الحل: التأكد من استدعاء `delete`/`delete[]` المناسب فور انتهاء معالجة كل طلب، أو الأفضل استخدام مؤشرات ذكية (`unique_ptr`/`shared_ptr`) بحيث تُدار الذاكرة تلقائياً مع خروج المؤشرات من نطاقها بعد كل طلب

### تمرين 3: جدول قرار — اختيار نوع المؤشر الذكي المناسب

**السيناريو:**
نظام إدارة ملفات يحتوي على: (أ) كائن ملف واحد يمتلكه تابع واحد فقط طوال حياته، (ب) كائن إعدادات مشترك يحتاج الوصول إليه عدة توابع مختلفة في آنٍ واحد.

**المطلوب:**
1. حدد لكل حالة نوع المؤشر الذكي المناسب واملأ الجدول التالي

**نموذج الحل:**
| الحالة | نوع المؤشر المناسب | السبب |
| --- | --- | --- |
| كائن الملف (مالك واحد فقط) | `unique_ptr` | ملكية حصرية وواضحة، لا حاجة لمشاركته |
| كائن الإعدادات (مشترك بين عدة توابع) | `shared_ptr` | يحتاج عدة أطراف الوصول لنفس الكائن في آنٍ واحد دون معرفة أيهم سينهي استخدامه أخيراً |

## الجزء الرابع: تمارين تتبع التنفيذ

### تمرين تتبع 1: حساب عناوين عناصر مصفوفة

**المدخل:**
```cpp
int a[4] = {5, 10, 15, 20};   // assume &a[0] = 0x2000, sizeof(int) = 4
```

**تتبّع خطوة بخطوة (أكمل الجدول):**
| الخطوة | العملية | العنوان/القيمة |
| --- | --- | --- |
| 1 | `&a[0]` | ؟ |
| 2 | `&a[1]` | ؟ |
| 3 | `*(a+2)` | ؟ |
| 4 | `&a[3]` | ؟ |

**نموذج الحل:**
| الخطوة | العملية | العنوان/القيمة |
| --- | --- | --- |
| 1 | `&a[0]` | `0x2000` |
| 2 | `&a[1]` | `0x2004` |
| 3 | `*(a+2)` | القيمة `15` (العنصر عند `0x2008`) |
| 4 | `&a[3]` | `0x200C` |

**النتيجة:** كل خانة تنتقل بمقدار 4 بايت بالضبط (حجم `int`)، وقيمة `*(a+2)` تساوي `a[2] = 15`.

### تمرين تتبع 2: تتبع عداد مراجع shared_ptr خطوة بخطوة

**المدخل:**
```cpp
shared_ptr<int> p1 = make_shared<int>(7);
shared_ptr<int> p2 = p1;
shared_ptr<int> p3 = p2;
p1.reset();
```

**تتبّع خطوة بخطوة (أكمل الجدول):**
| الخطوة | العملية | عداد المراجع |
| --- | --- | --- |
| 1 | إنشاء `p1` | ؟ |
| 2 | `p2 = p1` | ؟ |
| 3 | `p3 = p2` | ؟ |
| 4 | `p1.reset()` | ؟ |

**نموذج الحل:**
| الخطوة | العملية | عداد المراجع |
| --- | --- | --- |
| 1 | إنشاء `p1` | 1 |
| 2 | `p2 = p1` | 2 |
| 3 | `p3 = p2` | 3 |
| 4 | `p1.reset()` | 2 (يتحرر `p1` من الملكية فقط، الكائن لا يزال حياً بفضل `p2` و`p3`) |

**النتيجة:** الكائن يبقى حياً بعد `p1.reset()` طالما `p2` و`p3` لا يزالان يشيران إليه.

### تمرين تتبع 3: تتبع حالة مؤشرات unique_ptr بعد النقل

**المدخل:**
```cpp
unique_ptr<int> a = make_unique<int>(100);
unique_ptr<int> b = move(a);
unique_ptr<int> c = move(b);
```

**تتبّع خطوة بخطوة (أكمل الجدول):**
| الخطوة | العملية | حالة `a` | حالة `b` | حالة `c` |
| --- | --- | --- | --- | --- |
| 1 | إنشاء `a` | ؟ | ؟ | ؟ |
| 2 | `b = move(a)` | ؟ | ؟ | ؟ |
| 3 | `c = move(b)` | ؟ | ؟ | ؟ |

**نموذج الحل:**
| الخطوة | العملية | حالة `a` | حالة `b` | حالة `c` |
| --- | --- | --- | --- | --- |
| 1 | إنشاء `a` | يملك 100 | غير معرَّف بعد | غير معرَّف بعد |
| 2 | `b = move(a)` | فارغ (nullptr) | يملك 100 | غير معرَّف بعد |
| 3 | `c = move(b)` | فارغ | فارغ (nullptr) | يملك 100 |

**النتيجة:** المالك الحصري النهائي للقيمة `100` هو `c` فقط، بينما `a` و`b` كلاهما أصبحا فارغين.

## الجزء الرابع: بطاقات سؤال وجواب (Q&A Cards)

**Q1:** ماذا يمثل اسم المصفوفة في `C++‎` عند استخدامه كمؤشر؟
A: عنوان أول عنصر فيها (`&a[0]`).

**Q2:** بكم يتقدم المؤشر عند تنفيذ `p + 1`؟
A: بحجم النوع الذي يشير إليه (`sizeof(type)`)، وليس بايتاً واحداً.

**Q3:** ما الفرق بين `a[i]` و `*(a+i)`؟
A: لا يوجد فرق — كلاهما متكافئان تماماً وينتجان نفس النتيجة.

**Q4:** لماذا لا تُمرَّر المصفوفات بالقيمة للتوابع؟
A: لأن اسم المصفوفة هو عنوان، فيُمرَّر العنوان (عنوان أول عنصر) فقط.

**Q5:** ما الفرق بين الحجز الساكن والحجز الديناميكي؟
A: الساكن يُحدَّد وقت الترجمة (`int a[5]`)، والديناميكي وقت التشغيل (`new int[n]`).

**Q6:** ما الأداة المستخدمة لتحرير ذاكرة محجوزة بـ `new[]`؟
A: `delete[]` حصرياً، وليس `delete` وحدها.

**Q7:** ما هو `Memory Leak`؟
A: تسرب ذاكرة ناتج عن عدم تحرير ذاكرة محجوزة لم تعد مستخدمة.

**Q8:** ما هو `Dangling Pointer`؟
A: مؤشر يشير إلى ذاكرة تم تحريرها بالفعل عبر `delete`.

**Q9:** كيف تحل لغات مثل `Java` و`Python` مشكلة تسرب الذاكرة؟
A: عبر آلية `Garbage Collector` التي تُحرّر الذاكرة غير المستخدمة تلقائياً أثناء التشغيل.

**Q10:** كيف تحل لغة `Rust` مشكلة إدارة الذاكرة؟
A: عبر مفهوم `Ownership & Borrowing` الذي يُفرض أثناء الترجمة (Compile Time).

**Q11:** لماذا تبقى `C/C++/Rust` ضرورية رغم وجود لغات أسهل؟
A: لأنها تعمل مباشرة على العتاد دون الحاجة لبيئة تشغيل وسيطة، وهي أساس بناء أنظمة التشغيل والمتصفحات.

**Q12:** ما الفرق الجوهري بين `unique_ptr` و`shared_ptr`؟
A: `unique_ptr` مالك حصري غير قابل للنسخ، بينما `shared_ptr` يسمح بملكية مشتركة عبر عداد مراجع.

**Q13:** كيف يتم نقل ملكية `unique_ptr`؟
A: باستخدام `std::move`، وبعد النقل يصبح المؤشر الأصلي فارغاً.

**Q14:** ماذا يحدث لعداد المراجع عند نسخ `shared_ptr`؟
A: يزداد بمقدار 1.

**Q15:** ماذا يحدث للكائن المُدار بـ `shared_ptr` عندما يصل عداد المراجع للصفر؟
A: يتم حذفه وتحرير ذاكرته تلقائياً.

**Q16:** لماذا يُفضَّل `make_shared` على `shared_ptr<T>(new T())`؟
A: لأنه يخصص الكائن وكتلة التحكم في عملية تخصيص واحدة، مما يزيد الكفاءة.

**Q17:** ما ثغرة Heartbleed وما علاقتها بالمؤشرات؟
A: ثغرة أمنية في مكتبة `OpenSSL` نتجت عن عدم التحقق من حجم المصفوفة أثناء التنقل عليها بمؤشر.

**Q18:** هل تتحقق `C++‎` تلقائياً من تجاوز حدود المصفوفة؟
A: لا، على خلاف لغات مثل `Java` التي ترمي استثناءً (`IndexOutOfBoundException`).

## الجزء الخامس: كتابة الكود الكامل (مرجع شامل)

```cpp
#include<iostream>
#include <memory>
using namespace std;

int main(){
    // --- Part 1: Arrays as pointers ---
    int a[5];
    int *aPtr = a;
    cout << a << endl;
    cout << *aPtr << endl;
    cout << &a[0] << endl;

    int arr[5] = {1,2,3,4,5};
    for(int i=0; i<5; i++)
        cout << arr[i] << endl;
    for(int i=0; i<5; i++)
        cout << *(arr+i) << endl;

    // --- Part 2: Pointer used as array ---
    int big[50] = {1,2,3,4,5};
    int *b = big;
    for(int i=0; i<5; i++)
        cout << b[i] << endl;

    // --- Part 3: Dynamic memory allocation ---
    int n;
    cin >> n;
    int *dynArr = new int[n];
    for(int i=0; i<n; i++)
        cin >> dynArr[i];
    for(int i=0; i<n; i++)
        cout << dynArr[i];
    delete[] dynArr;
    dynArr = nullptr;

    // --- Part 4: new / delete for single object ---
    int* p1 = new int;
    cout << "Allocated P1" << endl;
    cout << "*p1=" << *p1 << endl << "p1=" << p1 << endl;
    delete p1;
    p1 = nullptr;

    int* p2 = new int[10];
    cout << "Allocated P2" << endl;
    cout << "*p2=" << *p2 << endl << "p2=" << p2 << endl;
    delete[] p2;
    p2 = nullptr;

    // --- Part 5: unique_ptr ---
    unique_ptr<int> ptr = make_unique<int>();
    *ptr = 5;
    cout << *ptr << endl;
    unique_ptr<int> x = move(ptr);
    cout << *x;

    // --- Part 6: shared_ptr ---
    shared_ptr<int> sptr = make_shared<int>();
    *sptr = 5;
    cout << *sptr << endl;
    shared_ptr<int> sx = sptr;
    cout << *sptr << endl;
    cout << *sx;

    return 0;
}
```

## الجزء الخامس: أسئلة نظرية متوقعة بالامتحان

### سؤال 1: ما معنى التعامل مع المصفوفة كمؤشر في C++‎؟
**نموذج الإجابة:**
1. التعريف: اسم المصفوفة يُقيَّم كعنوان أول عنصر فيها (`&a[0]`)
2. المكونات/الشروط: يتطلب فهم `Pointer Arithmetic` وحساب العناوين بحجم النوع
3. مثال: `int *aPtr = a;` يجعل `aPtr` يحمل عنوان `a[0]` مباشرة
4. متى نستخدم: عند تمرير المصفوفات للتوابع، أو عند التنقل عليها ببطء عبر مؤشرات لأغراض الأداء

### سؤال 2: اشرح مفهوم Pointer Arithmetic وأثره على العناوين
**نموذج الإجابة:**
1. التعريف: عملية إضافة/طرح رقم صحيح من مؤشر تنتج مؤشراً جديداً منزاحاً بعدد كامل من العناصر
2. المكونات/الشروط: يعتمد على `sizeof(type)` الذي يشير إليه المؤشر
3. مثال: لمصفوفة `int`، فإن `a+1` تنتقل 4 بايت وليس بايتاً واحداً
4. متى نستخدم: عند التنقل بين عناصر المصفوفات باستخدام المؤشرات بدلاً من الفهرسة العادية

### سؤال 3: ما الفرق بين الحجز الساكن والحجز الديناميكي للذاكرة؟
**نموذج الإجابة:**
1. التعريف: الساكن يُحدد وقت الترجمة، والديناميكي يُحدد وقت التشغيل
2. المكونات/الشروط: الساكن يستخدم `int a[N];`، الديناميكي يستخدم `new int[n];`
3. مثال: عندما لا نعرف حجم المصفوفة إلا بعد قراءته من المستخدم، نستخدم الحجز الديناميكي
4. متى نستخدم: الديناميكي عندما يكون الحجم متغيراً وقت التشغيل؛ الساكن عندما يكون الحجم ثابتاً ومعروفاً مسبقاً

### سؤال 4: اشرح مشكلة تسرب الذاكرة (Memory Leak) وكيفية تجنبها
**نموذج الإجابة:**
1. التعريف: عدم تحرير ذاكرة محجوزة ديناميكياً رغم عدم الحاجة إليها بعد الآن
2. المكونات/الشروط: يحدث عند نسيان استدعاء `delete`/`delete[]` المطابق لكل `new`/`new[]`
3. مثال: حجز مصفوفة داخل حلقة تكرارية دون تحريرها في كل دورة
4. متى نستخدم الحل: استخدام مؤشرات ذكية (`unique_ptr`/`shared_ptr`) لتفادي النسيان اليدوي كلياً

### سؤال 5: قارن بين آليات إدارة الذاكرة في C++‎ الكلاسيكية، Garbage Collector، وRust
**نموذج الإجابة:**
1. التعريف: ثلاث فلسفات مختلفة لإدارة الذاكرة
2. المكونات/الشروط: `C++‎` يدوية بالكامل (`new`/`delete`)، `Garbage Collector` آلية وقت التشغيل، `Rust` آلية وقت الترجمة (`Ownership`)
3. مثال: `Java`/`Python` تستخدم `Garbage Collector`؛ `Rust` تفرض قواعد الملكية أثناء الترجمة
4. متى نستخدم: `C++‎` عند الحاجة لأقصى تحكم وأداء؛ `Garbage Collector` عند أولوية سهولة التطوير؛ `Rust` عند الرغبة بأمان الذاكرة دون كلفة أداء وقت التشغيل

### سؤال 6: لماذا لا تزال لغات مثل C/C++/Rust ضرورية رغم وجود لغات أعلى مستوى؟
**نموذج الإجابة:**
1. التعريف: هذه اللغات تعمل مباشرة على العتاد دون الحاجة لبيئة تشغيل وسيطة
2. المكونات/الشروط: لا تحتاج `JVM`, `.Net`, متصفح، أو مفسر (Interpreter)
3. مثال: `Microsoft Office`, `Chrome`, `Firefox`, نواة `Android` (Linux Kernel) كلها مبنية بـ `C++‎`/`Rust`
4. متى نستخدم: عند بناء أنظمة التشغيل، البيئات التي تعمل ضمنها لغات أخرى، أو برامج تتطلب أداءً حرجاً للغاية

### سؤال 7: اشرح مفهوم unique_ptr وقاعدة عدم قابليته للنسخ
**نموذج الإجابة:**
1. التعريف: مؤشر ذكي يمتلك كائناً بشكل حصري ويحرره تلقائياً عند خروجه من النطاق
2. المكونات/الشروط: غير قابل للنسخ (Non-copyable)، لكنه قابل للنقل (Movable) عبر `std::move`
3. مثال: `unique_ptr<int> x = move(ptr);` — بعدها يصبح `ptr` فارغاً
4. متى نستخدم: عندما يكون هناك مالك حصري واضح للمورد طوال حياته دون حاجة للمشاركة

### سؤال 8: اشرح آلية عداد المراجع في shared_ptr
**نموذج الإجابة:**
1. التعريف: آلية داخلية (`Reference Counting`) تتبّع عدد المؤشرات المالكة لنفس الكائن
2. المكونات/الشروط: يزداد العداد عند النسخ، وينقص عند تدمير أي مؤشر مالك، ويُحذف الكائن عند وصول العداد للصفر
3. مثال: `shared_ptr<int> x = ptr;` يزيد العداد من 1 إلى 2
4. متى نستخدم: عندما تحتاج عدة أجزاء من البرنامج للوصول لنفس المورد دون معرفة أيها سينهي استخدامه أخيراً

### سؤال 9: ما العلاقة بين ثغرة Heartbleed ومفهوم المؤشرات في C++‎؟
**نموذج الإجابة:**
1. التعريف: Heartbleed ثغرة أمنية شهيرة ضمن مكتبة `OpenSSL`
2. المكونات/الشروط: نتجت عن عدم التحقق من حجم مصفوفة أثناء التنقل عليها بالاعتماد على مؤشر
3. مثال: قراءة بيانات خارج حدود المصفوفة المخصصة، ما كشف بيانات حساسة من ذاكرة الخادم
4. متى نستخدم الدرس المستفاد: التحقق دوماً من حدود المصفوفات عند استخدام المؤشرات، خصوصاً في الأنظمة الحساسة أمنياً

## الجزء السادس: ورقة المراجعة السريعة (Cheat Sheet)

### 🔑 خريطة العلاقات بين المحاضرات
| المحاضرة | ترتبط مع | كيف؟ |
| --- | --- | --- |
| المحاضرة 1 (مقدمة المؤشرات) | المحاضرة 2 (الحالية) | تبني المحاضرة 2 على أساسيات `&`, `*` من المحاضرة الأولى وتوسّعها للمصفوفات والحجز الديناميكي |
| المحاضرة 2 (الحالية) | المحاضرة 3 (القادمة: `const + pointers`, `pointers to functions`) | فهم المؤشرات والحجز الديناميكي هنا ضروري لفهم `const pointer` والمؤشرات على التوابع لاحقاً |

### 🔑 أهم النقاط الذهبية
| الموضوع | النقاط |
| --- | --- |
| المصفوفات كمؤشرات | اسم المصفوفة = عنوان أول عنصر؛ `a[i]` ≡ `*(a+i)` |
| حساب المؤشرات | الإزاحة تُحسب بوحدة `sizeof(type)` وليس بايتاً واحداً |
| الحجز الديناميكي | `new`/`new[]` لحجز الذاكرة وقت التشغيل بحجم متغير |
| تحرير الذاكرة | `delete`/`delete[]` يجب أن يطابقا أداة الحجز تماماً |
| Memory Leak | ينتج عن نسيان `delete`، يُهدر ذاكرة تراكمياً |
| المؤشرات الذكية | `unique_ptr` (حصري، غير قابل للنسخ) و`shared_ptr` (مشترك بعداد مراجع) |

### 🔑 مرجع سريع
| الرمز/المصطلح | المعنى | يُستخدم في |
| --- | --- | --- |
| `new T` | حجز كائن مفرد ديناميكياً | يُقابله `delete` |
| `new T[n]` | حجز مصفوفة ديناميكياً بحجم `n` | يُقابله `delete[]` |
| `make_unique<T>()` | إنشاء `unique_ptr` آمن | بديل موصى به عن `new` المباشر مع `unique_ptr` |
| `make_shared<T>()` | إنشاء `shared_ptr` بكفاءة | يخصص الكائن وكتلة التحكم معاً |
| `std::move` | نقل الملكية بين مؤشرين ذكيين | ضروري مع `unique_ptr` لعدم إمكانية النسخ |

### 🔑 قواعد ذهبية لا تُنسى
| # | القاعدة |
| --- | --- |
| 1 | كل `new` يقابله `delete`، وكل `new[]` يقابله `delete[]` — لا تخلط بينهما أبداً |
| 2 | `Pointer Arithmetic` يتحرك بوحدة "عنصر كامل" (`sizeof(type)`) وليس بايتاً واحداً |
| 3 | لا تعتمد أبداً على ترتيب المتحولات في الذاكرة (Stack layout) — سلوك غير معرّف يختلف حسب المصرّف |
| 4 | استخدم `unique_ptr` كخيار افتراضي، وانتقل إلى `shared_ptr` فقط عند الحاجة الفعلية للمشاركة |
| 5 | بعد `delete p;`، صفِّر المؤشر يدوياً (`p = nullptr;`) لتجنب استخدامه بالخطأ لاحقاً |
