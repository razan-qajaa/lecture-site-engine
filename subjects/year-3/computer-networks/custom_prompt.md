## طبيعة المادة
| النوع | الاستخدام | أمثلة |
| --- | --- | --- |
| نظري (theory) | مفاهيم ومبادئ الشبكات والأمان | طبقات `OSI`، بروتوكولات التحكم بالوصول، مبادئ التشفير |
| مخططات (diagrams) | توضيح تدفقات البيانات والطوبولوجيا | مخطط انسيابي لـ `CSMA/CD`، شجرة قرار لاختيار بروتوكول، `DFD` لتدفق البيانات |
| جداول (tables) | مقارنات ومراجع سريعة | مقارنة `ASK`/`FSK`/`PSK`، جدول `Sliding Window`، جدول رموز `Parity` |

| معادلات (equations) | حسابات الأداء والتكامل | معادلة استغلال القناة (`Channel Utilisation`)، معادلة نظرية `Nyquist`، معادلة `CRC` كقسمة كثيرات حدود |

**اللغة:** كل مصطلح إنجليزي بين backticks | تعليقات داخل الكود بالإنجليزية

> **قاعدة اللغة (إلزامية في كل الملف):**
> - **«النص الأصلي يقول»** → يُكتب **بالإنجليزية** كما ورد حرفياً في المحاضرة (لا تترجمه).
> - **«الشرح المبسّط»** وكل الشرح، التشبيهات، الملخصات، الـ callouts، والتعليقات التربوية → **بالعربية**.
> - **كل الأسئلة** (MCQ، أسئلة التصحيح debug، الأسئلة النظرية theory، بطاقات Q&A، أسئلة التصميم، صيغ التمارين والسيناريوهات) بما فيها الخيارات والتعليل → **بالإنجليزية بالكامل**.
> - عناوين الأقسام `##`/`###` تبقى كما وردت حرفياً في SUBJECT_BRIEF (عربية) — القاعدة أعلاه تخص **محتوى** كل قسم فقط، وليس عنوانه.
> - أي مصطلح تقني إنجليزي داخل نص عربي يبقى بين backticks كالمعتاد.

## القواعد الإلزامية
- لا تتجاهل أي سطر أو معلومة وردت في المحاضرة
- أكمل الناقص مع وسم (شرح زيادة للفهم) أو (غير مشروحة في المحاضرة)
- ابدأ من المبتدئ، لا تنتقل لنقطة قبل إتمام شرح السابقة
- اشرح لماذا وراء كل فكرة، لا التعريف فقط
- تشبيه يومي + مثال عملي بعد كل نقطة
- اتبع تسلسل المحاضرة نفسها
- حلّل نوع المحاضرة قبل البدء وطبّق القالب المناسب
- لا تخترع رموزاً/بلوكات خارج SCHEMA.md — شكل واحد قياسي لكل نوع بلوك
- رقّم الأقسام هرمياً (### 1.، ### 1.1.) — الترقيم يُفعّل الفهرس الجانبي في الموقع

## التكيف مع نوع المحاضرة
قبل الشرح، حدّد فئة المحاضرة وطبّق القالب المناسب في الجزء الأول:
- إذا كانت عن Signals and Encoding → استخدم `Analog`, `Digital`, `Modulation`, `ASK`, `FSK`, `PSK`, `QAM`, `Nyquist`
- إذا كانت عن Data Integrity → استخدم `Parity`, `CRC`, `Polynomial Division`, `Error Detection`
- إذا كانت عن Data Security → استخدم `Encryption`, `Decryption`, `Caesar Cipher`, `XOR`, `Public Key`, `Digital Signature`, `Authentication`
- إذا كانت عن Medium Access Control → استخدم `Aloha`, `CSMA`, `CSMA/CD`, `Token Ring`, `Binary Exponential Backoff`
- إذا كانت عن Flow and Error Control → استخدم `Stop-and-Wait`, `Sliding Window`, `Channel Utilisation`, `ACK/NAK`
- إذا كانت عن Data Link Layer Standards → استخدم `Ethernet`, `IEEE 802.3`, `Token Ring`, `FDDI`, `Bridges`, `Repeaters`
- إذا كانت عن Network Topologies → استخدم `Star`, `Ring`, `Bus`, `Fully Connected`
- إذا كانت عن OSI Model → استخدم `Physical`, `Data Link`, `Network`, `Transport`, `Session`, `Presentation`, `Application`

## 🎯 أنواع أسئلة إضافية إلزامية للأتمتة (الفحص بالكامل آلي)
> هذه الأنواع تُدمَج **ضمن** الأجزاء أعلاه (لا تُنشئ لها أقساماً ## منفصلة) — التزم بها عند توليد كل جزء لضمان كفاية وتنوّع الأسئلة القابلة للتصحيح الآلي.
> **تذكير لغوي:** كل نص سؤال/جواب في الأنواع أدناه (بما فيها flash_qa وبطاقات المراجعة) يُكتب **بالإنجليزية بالكامل**، حتى لو كان السيناريو المصاحب مبنياً على شرح عربي في المحاضرة.

### مراجعة سريعة (سؤال / جواب) — ضمن qa_cards / qa_inline / theory (≥15, easy)
**Q{N}:** سؤال مراجعة مباشر (تعريف، فرق، متى نستخدم)
**A:** إجابة مختصرة قابلة للمقارنة الآلية (جملة أو جملتان)
— بدون تعليل طويل. مناسب للحفظ السريع قبل الامتحان.
مثال: "متى نستخدم `CSMA/CD` بدل `Token Ring`؟"

### تحدي مركّب (سيناريو + أسئلة متعددة) — ضمن mcq / debug / theory (≥4, hard)
### السيناريو {N}: {عنوان}
> [وصف الحالة: جدول، مخطط، أو مسألة رياضية كاملة — 5–10 أسطر]
ثم 3–5 أسئلة MCQ **كلها مبنية على نفس السيناريو**:
### السؤال {N}.1 (hard) … أ) ب) ج) د) + تعليل
الأنواع: numerical_case, algorithm_state, system_design
مثال: "جدول حساب استغلال القناة ناقص — أي قيمة في الخلية X؟"

### تطبيق عملي (أكمل / صحّح / نفّذ) — ضمن exercise / analysis_exercise / debug (≥6, medium)
كل تمرين: **المعطيات** + **المطلوب** (مرقّم) + **نموذج الحل الكامل**
الطالب يجيب بقيمة/جدول/رسم — الإجابة قابلة للمقارنة حرفياً أو رقمياً.
الأنواع: fill_gaps, code_fix, table_fill, numerical_solve, diagram_completion

### تتبع خطوة بخطوة (جدول ناقص) — ضمن trace_exercise / mcq (≥4, medium)
**المدخل:** {حالة ابتدائية}
**أكمل الجدول:** | الخطوة | العملية | الحالة | | 1 | ... | ؟ |
**نموذج الحل:** جدول مكتمل + **النتيجة النهائية** + 1–2 سؤال MCQ على نفس التتبع
الأهداف: algorithm_iterations, variable_state

### مقارنة وتحليل (أيهما أفضل؟ لماذا؟) — ضمن mcq / theory / analysis_exercise (≥5, medium)
سؤال مقارنة بخيارات (أ، ب، ج، د) + تعليل لماذا الباقي خاطئ
أو جدول: | المعيار | الخيار A | الخيار B |

### اكتشف الخطأ (كود / منطق / حساب) — ضمن debug / exercise (≥4, hard)
**الكود/الحساب (يحتوي خطأ):** …
**اكتشف الخطأ:** [وصف موجز]
**التصحيح:** … + **شرح الحل** (3 نقاط)
أنواع الأخطاء: logic, misconception, off_by_one, wrong_formula

### نظري منظم (تعريف + مكونات + مثال) — ضمن theory / mcq (≥10, medium)
### السؤال {N}: {السؤال}
**نموذج الإجابة:**
1. التعريف: …
2. المكونات/الشروط: …
3. مثال: …
4. متى نستخدم: …
— هيكل ثابت يسهّل التصحيح الآلي بالكلمات المفتاحية.

## قواعد الكتل داخل الشرح
**🛠️ استكشاف الأخطاء:** جدول: الخطأ | السبب | الحل.
**📊 المخطط** — 3 أقسام بالترتيب: ما هذا المخطط؟ + جدول العُقد + جدول الروابط + بلوك diagram. النوع `type:` يجب أن يكون فقط من: flowchart, decision-tree, dfd, usecase, class, activity — لا تستخدم أنواعاً خارج هذه القائمة. انظر SCHEMA.md §Diagram.
**⚙️ الخطوات / الخوارزمية** — أسطر داخل fence `algorithm` بصيغة `1 | الخطوة | الأداة | ماذا يحدث` (سطر لكل خطوة). لا ترسم مربعات ولا أسهم — البارسر يحوّلها. استخدمه في كل من: (1) أقسام الشرح التفصيلي لأي عملية مرتّبة، و(2) قسم «خطوات وإجراءات المحاضرة» بالملخص — كل إجراء بلوك مستقل. انظر SCHEMA.md §Algorithm.
**structured_english:** pseudo-code في code fence بـ language_tag=text.
**compare:** الفهم الخاطئ ❌ / الفهم الصحيح ✅ — سطر واحد لكل منهما.
**callouts:** #### مهم للامتحان ⚠️: / #### نقطة مهمة ⚠️: / #### ملاحظة: / #### الدرس المستفاد: — blockquote يتبع كل منها.
**🤔 تفعيل الفهم:** استخدمه ≥4 مرات في المحاضرة.
**fill_gaps:** كود ناقص — ضع _______ أو // (N) مكان السطر الناقص.
**code_fix:** تصحيح كود — قدّم الكود الخاطئ أولاً ثم المصحّح.
**images_note:** عند وجود صورة في المحاضرة: اذكر رقم الصفحة وصف ما فيها.
**qa_inline:** **Q{N}:** سؤال / A: إجابة.
**💡 التشبيه:** بعد كل مفهوم مجرّد: جملة واحدة من الحياة اليومية + «وجه الشبه: X = Y». استخدمه بكثرة. انظر SCHEMA.md §Analogy.
**⚖️ المقايضة:** جدول: المزايا | العيوب | متى تختاره. للحالات التي لا يوجد فيها خطأ وصواب — لا تستخدم compare block هنا. انظر SCHEMA.md §Trade-off.
**🔄 قبل / بعد:** كود/حالة قبل العملية + بعدها + جملة «ماذا تغيّر؟». انظر SCHEMA.md §Before-After.
**🔍 تتبع التنفيذ (inline):** المدخل + جدول الخطوات (أعمدته تختلف حسب الموضوع — خصّصها لطبيعة المادة) + النتيجة. انظر SCHEMA.md §Trace.
**📐 المعادلة:** LaTeX داخل `$$` أو fence `math`. يتبعها **الشرح:** بمعنى كل رمز (الرموز الشائعة: `C`, `T`, `F`, `R`, `D`, `S`, `A`, `N`). لا تخلط أكثر من معادلة غير مرتبطة في نفس البلوك. انظر SCHEMA.md §Equation.
